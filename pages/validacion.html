¿Cómo validamos el código fuente de una aplicación?

Aspectos a considerar y formas de realizarlo

Validar desde los cimientos

Validar el código fuente de una aplicación, será sumamente relevante para garantizar la calidad del producto final, puesto que, al probar el código escrito por los desarrolladores, tendremos más chances de que existan menos defectos, que, a su vez, se conviertan en fallas en el sistema.

Algunos aspectos a considerar para validar el código fuente, son los siguientes:

Cumplir estándares de codificación: Como analista QA, muchas veces, nuestra labor estará sujeta a diferentes pautas y estándares que se deben cumplir, los cuales varían según el lenguaje de programación utilizado, un ejemplo de ello es Java, que contiene pautas dirigidas por Oracle y Google.

Funcionalidad: Podemos asegurarnos de que el código cumpla con los requerimientos establecidos, a través de pruebas exhaustivas y revisiones de casos de prueba, entregando reportes oportunos a todo el equipo involucrado.

Seguridad: El QA puede verificar que el código sea seguro y esté protegido contra posibles vulnerabilidades, como algunas propuestas por el proyecto Open Web Application Security Project (OWASP), que determina una serie de causas que pueden llevar a un software a ser inseguro.

Normativas y regulaciones: Al momento de desarrollar una aplicación, esta podrá atenerse al cumplimiento de regulaciones específicas, que aseguren aspectos legales de acuerdo al país o rubro.

Cobertura de pruebas: Asegurar un número suficiente de pruebas unitarias, aumentará la confianza en el producto. Para ello, el QA debe revisar que la cobertura sea lo suficientemente grande para cubrir la mayor cantidad de código fuente.

¿De qué formas vamos a validar el código fuente?

Análisis estático: La pruebas estáticas significaran una revisión manual o mediante herramientas automatizadas del código fuente sin ejecutar. Al analizar el código fuente, nos entregará información sobre la profundidad de anidamiento del código, la complejidad ciclomática, estándares de codificación, flujo de control, relaciones de datos y número de rutas entre línea de código.

Pruebas unitarias: El QA puede realizar una revisión exhaustiva de las pruebas unitarias existentes, para asegurar la cobertura adecuada del código. Con esto, se podrá ayudar a los desarrolladores a identificar áreas que requieran pruebas adicionales.

Pruebas no funcionales: Conocidas como pruebas de las características de software, evalúan el correcto funcionamiento y velocidad del producto, como el tiempo de respuesta, seguridad, carga, etc. Algunas pruebas que encontraremos en esta clasificación son: pruebas de rendimiento, pruebas de estrés, pruebas de usabilidad, pruebas de confiabilidad y pruebas de portabilidad.

Pruebas Automatizadas: Al escribir scripts de prueba, podremos validar el código en repetidas ocasiones, facilitando la detección temprana de defectos en el código.

Pruebas funcionales: Responden a lo que debería hacer el sistema, obteniendo una cobertura funcional, que radica en cuantos componentes del sistema han sido probados. Algunos ejemplos de pruebas funcionales, serían las pruebas de caja negra y caja blanca.

Pruebas de caja negra: Aquellas pruebas en que no se requiere saber cómo funciona el software, sino los resultados que se obtienen al probarlo. Aquí encontraremos las pruebas de partición de equivalencia, análisis de valores frontera, pruebas de tabla de decisión, pruebas de transición de estado y pruebas de caso de uso.

Pruebas de caja blanca: Se basan en el código fuente del software y requieren saber cómo es su funcionamiento. Aquí encontraremos las pruebas de cobertura y sentencia, y pruebas de cobertura y decisión.

Es importante conocer y probar el código fuente del proyecto en cuestión, debido a que como Analista QA, encontrarás defectos y fallas que propiciaran información crucial para disminuir los costes y esfuerzos que significan reparar bugs en etapas más tardías de desarrollo.
